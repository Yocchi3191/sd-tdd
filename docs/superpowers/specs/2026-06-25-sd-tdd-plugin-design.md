# sd-tdd プラグイン設計

日付: 2026-06-25

## 背景・課題

仕様駆動開発（SDD）は「仕様」を作るが、仕様自体はコードを物理的に守らない。仕様とコードは別の生き物で、仕様だけ更新を忘れても何も壊れない（静かに陳腐化する）。一方TDDのテストは、壊れたり失敗することでコードを守り、かつテストが古くなった（前提が崩れた）ことを失敗という形で知らせてくれる。

このプラグインは「仕様をちゃんと作り、ちゃんと作った仕様からちゃんとテストを作り、テストが実装のゴールとして機能する」状態を、Claude Codeのワークフローとして強制することを目的とする。

現状の開発フローは「ふわっとした機能要望 → issue起票skillが概要を作る → そのまま実装」であり、仕様を詰める工程・仕様からテストを導出する工程が存在しない。

## コア思想

**持続的な「仕様書」を一切持たない。**

仕様は「詰める」フェーズで一度だけ構造化されたリスト（REQ-1, REQ-2, ...）になり、テストに変換された時点で実質的に役目を終える。以後、生きた契約はテストだけが担う。

二層構造で考える：

| | issueのREQリスト（台帳 / ledger） | テストコード（契約 / contract） |
|---|---|---|
| 役割 | いつ・何を・なぜ正しいと思ったかの歴史的記録 | 現在の正しい振る舞いを強制する、生きた仕組み |
| 変更可否 | 既存エントリの文言は不変。誤りや競合は新規エントリで「supersede」を宣言する（ADRの「古い決定は編集せず新しいADRで上書きする」と同じパターン） | 自由に編集・削除してよい。git履行が監査ログになる |

仕様書を持たない代わりに台帳（issue）と契約（テスト）に役割を分離することで、「仕様書の二重管理・更新漏れ」という本来の問題を構造的に消す。

## 全体フロー

```
ふわっとした機能要望
      │
      ▼
[1. spec-interview]  ── 対話で要求を詰め、REQ-1, REQ-2... の検証可能なリストにする
      │                  → issueに記録（台帳の初版）
      ▼
[2. spec-to-tests]   ── 各REQ-IDに対応する失敗するテストを生成
      │                  → it("REQ-1_〇〇のときXXする", ...) 形式
      ▼
[3. coverage-check]  ── REQリストとテストのREQ-ID対応を機械的に検証
      │                  漏れ・対応なしテストを報告。通るまで1↔2を往復
      │                  （1に戻る場合も既存REQの書き換えはせず、新規REQを追記する）
      ▼
[4. 実装]            ── sd-tddの責務外。superpowersに依存し
                         test-driven-development skillに引き継ぐ
```

実装フェーズ（4）はsd-tdd自体のスキルとしては作らない。`plugin.json`で`superpowers`への依存を宣言し、失敗するテスト群ができた時点でsuperpowersの`test-driven-development`skillに自然に引き継ぐ。車輪の再発明を避ける。

```json
{
  "name": "sd-tdd",
  "dependencies": [
    { "name": "superpowers", "version": "^6.0.0" }
  ]
}
```

## コンポーネント

### 1. `spec-interview` skill

- **入力**: 「こんな機能を作ってほしい」という自然言語の要望
- **振る舞い**: brainstormingスキルと同様に、一問一答で要求を詰める対話を行う。ただし出力形式が異なり、自由記述の設計書ではなく、検証可能な要求を1項目1事実の粒度で列挙する
- **出力**: issue本体に番号付きREQリストを記録する

```
REQ-1: ユーザーが空文字を送信したら400を返す
REQ-2: リトライは3回まで — vendor APIのレート制限による
```

- 理由（なぜ）の記載は原則テスト側（後述）に委ねるが、以下のいずれかに当たる場合はREQ項目に直接書く：
  1. 検討した代替案の説明が必要
  2. コードから推測できない外部要因（要望・規制・インシデント等）がある
  3. 一文で言い切れない

### 2. `spec-to-tests` skill

- **入力**: issueのREQリスト
- **振る舞い**: 各REQ-IDについて、プロジェクトの既存テストフレームワーク・規約を検出し、それに沿った失敗するテストを生成する。1つのREQに対し複数のテスト（異常系・正常系など）があってよい
- **命名規則**: テスト名にREQ-IDを埋め込む。例（vitest）:

```ts
it("REQ-1_空文字を送信したら400を返す", () => { ... })
```

- 理由が一文かつコード/ドメインの事実だけで言えるなら、テスト名やコメントに埋め込む（B方針）。前述の3条件に当たる場合はREQ項目側（A方針）に委ね、テスト側には軽いポインタ（issue番号など）だけ残す
- 生成したテストは実装前に一度実行し、「正しい理由でred」になっていることを確認する（タイポ等でエラー落ちしていないこと）

### 3. `coverage-check` skill

- **入力**: issueのREQリスト + 生成されたテストファイル
- **振る舞い**: テストファイルからREQ-IDをグレップし、issueのREQリストと突き合わせる、機械的でLLM判断に依存しない検証
  - **漏れ**: テストが1つも無いREQ-ID → ブロックし、spec-to-testsに戻って生成を促す
  - **スコープ外**: issueに無いREQ-IDを持つテスト → 警告。意図的な追加なら台帳に追記してもらう
- このチェックは「一度きりの儀式」ではない。実装中に新しいエッジケースが見つかってREQが追記された場合、その都度re-runできる軽量な確認である

### ledgerの追記・supersedeルール（コンポーネント共通）

- 既存REQ項目の文言は**書き換えない**
- 実装中に新しい要求/エッジケースが見つかったら、新規REQ-N+1として追記し、対応するテストを書く
- 過去のREQが誤っていた・他issueのREQと競合した場合も、該当項目を書き換えず、新規エントリで「`[superseded by REQ-N]`」のように上書きを宣言する。テスト側は新しい正しい振る舞いに合わせて自由に書き換える（テストは契約であって台帳ではないため）

## 既存スキル・プラグインとの関係

- `personal-issue-management`などの既存skillとの統合は考慮しない。完全に独立したプラグインとして動作する
- `superpowers`にのみ`plugin.json`の`dependencies`で依存し、実装フェーズの`test-driven-development`skillを利用する

## スコープ外

- issueのクローズ処理、実装中の意思決定記録など、issueライフサイクル全体の管理
- 仕様書ドキュメントの作成・維持（本設計の核心として意図的に持たない）
