# dev-dojo プラグイン 設計

日付: 2026-07-03

## 背景・目的

ユーザー自身の要件定義・ソフトウェア設計・テスト設計の「センス」を鍛えたい、という修行目的の要望。既存の`sd-tdd`プラグインはスペック駆動TDDのワークフロー(REQ台帳生成→テスト生成→カバレッジ検証)を担うものであり、「人間の判断力そのものを鍛える」役割とは性質が異なる。1マーケットプレイスから複数プラグインを出すこと自体は自然なため、本リポジトリ内に新規プラグイン`dev-dojo`として追加する。

## 目的

1. 要件定義・ソフトウェア設計・テスト設計の意思決定の直前に割り込み、ユーザー自身に先に考えを言語化させる
2. ソクラテス式の問い返しでユーザーの論拠を鍛える(即座に正解を教えない)
3. どのプロジェクトで作業していても、実務タスクの中で自然に発火する

## 進め方についての補足

本プラグイン追加はsd-tddマーケットプレイス自体のメタ開発であり、過去の同種の作業(`sd-tdd:run`オーケストレーターskill追加、`task-filing`skill新設)と同様、`spec-interview`によるREQ台帳インタビューはやり直さず、`brainstorming`で確定した本設計をそのまま`writing-plans → 実装`に引き継ぐ。

ただし「1 issue = 1 タスク」の運用は維持する。ユーザー指示により、本設計の内容を`task-filing`の「file a new task」操作でそのままissue化してからタスクに着手する(`spec-interview`のインタビュー自体はスキップするが、issue起票は省略しない)。

## 全体構成

新規プラグイン`dev-dojo`を`plugins/dev-dojo/`に追加し、`.claude-plugin/marketplace.json`に2つ目のplugin エントリとして登録する。

- `plugins/dev-dojo/.claude-plugin/plugin.json`: `sd-tdd`のplugin.jsonを踏襲した最小構成。`dependencies`は持たない — 各skillはプロンプトのみで完結し、特定の他skillが無いと動作できないという機械的依存ではないため。
- コード資産(scripts)を持たないため`package.json`は作らない。
- 3つのskillを配置し、各`SKILL.md`は他ファイルへの参照を持たず単体で完結させる(進行ロジックの重複を許容 — sd-tddの既存5skillも各々SKILL.md単体で完結する設計であり、それに倣う)。

```
plugins/dev-dojo/
  .claude-plugin/
    plugin.json
  skills/
    requirement-mentor/
      SKILL.md
    design-mentor/
      SKILL.md
    test-design-mentor/
      SKILL.md
```

## 利用シーン・発火範囲

- **実務タスクへの介入型**を採用する。独立した「修行モード」(架空のお題を渡して添削する形)は採用しない。
- **あらゆるプロジェクト共通**で発火する。`dev-dojo`はこのリポジトリ固有ではなく、グローバルにインストールして使う汎用プラグインとして設計する。
- 各skillの`description`は、対象領域の意思決定の直前かつ「有意な・後戻りコストのある判断」に限定する文言を含める。変数名などの些末で可逆的な判断では発火させない。

## 各skill共通の進行ロジック

3つのskillはいずれも次の4ステップ構造を持つ。内容は各`SKILL.md`にそれぞれ自己完結で記述する(多少のプローズの重複は許容する)。

1. **発火**: 対象領域の意思決定の直前。些末・可逆的な判断は対象外。
2. **先に言わせる**: Claudeは自分の見解・提案を一切開示せず、「あなたならどう考える？根拠は？」とユーザーに問う。
3. **ソクラテス式の評価**: 領域ごとのルーブリックに沿って論拠を吟味する。弱ければ、見落としているケース・検討していない代替案・根拠の欠如を具体的に指摘し、再度問い直す。これを論拠が尽きるまで繰り返す。妥当な論拠が示されたら「なぜ納得したか」を一言添えて先に進む。
4. **降参ハンドリング**: ユーザーが「降参」「ヒントください」等、明示的に協力を求めたら、師匠は自分の見解とその理由を共有し、以降は協力モードに切り替えて一緒に答えを組み立てる。無限ループを避けるセーフティネットとして機能する。
   - いずれの終わり方(論拠が稴いだ/降参)でも、通常のフロー(`brainstorming`/`spec-interview`/TDDなど)に進む前に一言だけ振り返りコメントを残してから引き継ぐ。

**キャラクター性**: 厳しめのソクラテス式。即答で正解を教えず、「なぜそう言える？」「他の選択肢は検討したか？」と孤軍し、論拠の粗さを平然と突く。ただし乱暴ではなく丁寧語。

## 各skillの発火条件とルーブリック

### requirement-mentor

- **発火**: 要件・スコープ・受け入れ条件を確定させる直前(REQ台帳作成、PRD記述、issueのスコープ確定など)
- **ルーブリック**:
  - 反証可能か(テストで白黒つけられるか)
  - 「and/or」で複数の要求が1文に混ざっていないか
  - 曖昧語(「うまく」「適切に」「いい感じに」)を使っていないか
  - 誰のどんなニーズに根ざしているか説明できるか
  - 異常系・エッジケースを無視していないか
  - スコープ外が明記されているか

### design-mentor

- **発火**: アーキテクチャ・モジュール分割・インターフェース・データフローなど、ソフトウェア設計判断をする直前
- **ルーブリック**:
  - 単一責任になっているか
  - 疎結合・高凝集か
  - 代替案を最低2つ検討し、却下理由を言えるか
  - YAGNI(将来の仮定のために作っていないか)
  - 依存の向き(誰が誰に依存するか)を説明できるか
  - この設計はテストしやすいか

### test-design-mentor

- **発火**: テストケース・テスト設計(何をどうテストするか)を書く直前。TDDのred phase設計を含む
- **ルーブリック**:
  - 1テスト1振る舞いになっているか
  - 境界値・同値分割を考慮したか
  - 「なぜ今このテストが失敗するはずか」を説明できるか(red phaseの正当性)
  - 実装の詳細でなく振る舞いをテストしているか
  - 正常系だけでなく異常系・エッジケースも含むか
  - テスト名から意図が読み取れるか

## スコープ外

- 独立修行モード(架空のお題ライブラリを用意して単独練習させる形)の新設 — 今回は実務介入型を採用したため見送り
- 進捗ログ・採点履歴の永続化 — YAGNI、要望なし
- `sd-tdd`や`superpowers`側の既存SKILL.mdへの変更 — 疎結合を維持し、一切変更しない
- このリポジトリの`CLAUDE.md`更新 — `dev-dojo`はどのプロジェクトでも使う汎用プラグインであり、このリポジトリ固有の作業ルールではないため不要

## 検証方法

コード資産を持たないため自動テスト・mutation testingの対象にはならない。実装後、このリポジトリ上で小さな架空の設計判断(例: 「ログイン機能のrate limitをどう設計するか」)を1つ想定し、3skillそれぞれについて次を手動でドライラン確認する。

- 発火条件が意図通りのタイミングで反応するか
- 「先に言わせる」→「弱い論拠への反証」→「妥当な論拠なら先に進む」の流れが機能するか
- 「降参」と発話した際に協力モードへ正しく切り替わるか
